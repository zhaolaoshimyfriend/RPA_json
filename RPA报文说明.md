# RPA报文说明

> 本文档面向 **RPA 开发者** 和 **平台开发者**，用于明确报文的整体结构、任务创建与执行时的参数约定，以及各任务类型的报文使用方式。

---

## 一、文档目标

1. **报文整体结构**：统一描述任一任务所涉及的报文格式
2. **任务创建参数**：平台创建任务时需传入的报文结构
3. **任务执行结果**：RPA 执行完成后返回的报文结构
4. **任务使用方式**：不同任务类型如何组合和使用各类报文

---

## 二、报文整体结构

### 2.1 通用 envelope

所有任务相关的报文均遵循统一的顶层结构（以下为约定，具体实现可按需补充）：

```
{
  "taskType": "<任务类型标识>",     // 任务类型，如：login、taxInfoCollection、stampTaxDeclaration 等
  "taskId": "<任务实例ID>",          // 平台下发的任务唯一标识
  "loginInfo": { ... },             // 登录信息（公共，多数任务需携带）
  "taskContext": { ... },           // 任务上下文（公共，如适用）
  "workflowSpec": [ ... ],          // 工作流规格（公共，如适用）
  "业务参数": { ... },              // 业务参数，由任务类型决定
  "result": { ... }                 // 任务执行结果（RPA→平台，执行完成后填充）
}
```

### 2.2 创建时报文

平台创建任务时，在报文体顶层直接挂载执行所需参数（无 `params` 包装层）。主要包含：

- **公共参数**：如 `loginInfo`（登录信息），多数任务需携带
- **业务参数**：由具体任务类型决定，如税种、税款所属期、申报表数据等

### 2.3 执行后报文（result）

RPA 执行完成后，通过 `result` 返回执行结果。约定结构（可按实际实现补充）：

```
{
  "success": true | false,          // 是否执行成功
  "message": "<简要说明>",           // 成功/失败提示
  "data": { ... },                  // 业务数据，如采集结果、申报回执等
  "attachments": [ ... ],           // 附件（截图、回单等），如有
  "traceId": "<链路追踪ID>"          // 可选，用于问题排查
}
```

## 三、任务创建参数

不同任务类型的参数由本目录下对应子目录的 JSON 文件定义，在报文体顶层直接挂载。

| 任务类型 | 报文文件/路径 | 主要参数说明 |
|----------|---------------|--------------|
| 登录 | `公共参数/loginInfo.json` | 网报区、登录平台、代理登录账号、被代理企业等 |
| 涉税信息采集 | `涉税信息采集/rpaMessageConfig.json`、`taxInfoCollection.json` | 合并 `loginInfo` + `taxInfoCollection`，采集纳税人登记、资格、税费种认定等 |
| 印花税（季报） | `印花税税源采集和申报（季报）/` | （待补充） |
| 企业所得税预缴（A类） | `企业所得税预缴（A类）/` | （待补充） |
| 增值税及附加税 | `增值税及附加税（一般/小规模纳税人）/` | （待补充） |
| 社会保险费（青岛） | `社会保险费（青岛）/` | （待补充） |
| 财务报表季报 | `财务报表季报（小企业会计准则）/` | （待补充） |
| 涉税信息同步 | `涉税信息同步/` | （待补充） |
| 申报核查 | `申报核查/` | （待补充） |

> 具体字段含义见各 JSON 中的 `_fieldRemarks`。

## 四、任务执行结果（result）

RPA 执行任务后，需按约定格式返回 `result`。各任务类型的 `data` 结构不同：

| 任务类型 | 成功时 data 典型内容 | 失败时处理 |
|----------|----------------------|------------|
| 登录 | 登录状态、会话信息（如有） | `success: false`，`message` 说明失败原因 |
| 涉税信息采集 | 采集到的涉税信息 JSON（结构可参考 `taxInfoCollection.json`） | 同上 |
| 申报类任务 | 申报回执、流水号、是否申报成功等 | 同上 |
| 核查类任务 | 核查结论、截图或证据列表 | 同上 |

> 各任务的具体 `result.data` 结构将随实现逐步补充完善。

## 五、不同任务如何引用报文

### 5.1 引用关系

```
任务类型 A（如：印花税申报）
  └─ loginInfo（公共） + 印花税申报业务参数，顶层挂载
  └─ result.data: 申报回执等

任务类型 B（如：涉税信息采集）
  └─ loginInfo（公共） + taxInfoCollection 业务参数，顶层挂载
  └─ result.data: 采集到的涉税信息
```

### 5.2 使用步骤

1. **平台侧**：根据 `taskType` 选择对应参数结构，从本目录 JSON 中组装，在报文体顶层挂载
2. **RPA 侧**：按约定解析顶层字段，执行具体操作
3. **RPA 侧**：执行完毕后，按约定结构填充 `result` 返回平台
4. **平台侧**：解析 `result`，更新任务状态、落库、通知用户等

### 5.3 公共依赖

多数任务依赖 `loginInfo`，因此：

- 可先执行登录任务获取会话
- 或由平台在创建任务时直接传入 `loginInfo`，供 RPA 在任务内完成登录

## 六、报文约定说明

### 6.1 `_fieldRemarks`

各 JSON 报文中的 `_fieldRemarks` 为字段说明，**仅用于文档与展示**，不参与业务逻辑。实现时请只解析业务字段。

### 6.2 目录与文件

- `公共参数/`：公共参数定义（如 `loginInfo.json`、`taskContext.json`）
- `涉税信息采集/`：涉税信息采集相关报文
- 其他子目录：按业务模块划分，对应具体任务类型
